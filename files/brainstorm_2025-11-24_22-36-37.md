## The Refined Vision: Ultrathink as "Git for Ideas"

Think of Ultrathink as **version control + CI/CD for your creative output**.

### Core Concept

Every idea is a **Project** that moves through states:

```
CAPTURED ‚Üí SCOPED ‚Üí IN_PROGRESS ‚Üí REVIEW ‚Üí SHIPPED ‚Üí LIVE
```

Ultrathink's job:
1. **Track state** of every project
2. **Enforce constraints** (max 3 active, minimum 1 shipped/week)
3. **Provide interfaces** to capture, work on, and ship
4. **Integrate with your tools** (not replace them)
5. **Publish to multiple destinations** with adapters

You stay in control. The system forces progress.

## Architecture: Simple, Modular, Extensible

### 1. Core: Project State Machine

**Data Model:**

```typescript
Project {
  id: string
  title: string
  capturedAt: timestamp

  // Classification
  type: 'content' | 'code' | 'research' | 'experiment'
  tags: string[]

  // Prioritization
  impactScore: 1-5
  effortScore: 'hours' | 'days' | 'weeks'
  priority: number (auto-calculated)

  // State
  status: 'captured' | 'scoped' | 'active' | 'review' | 'shipped' | 'live' | 'archived'

  // Content
  originalCapture: string (url, note, whatever)
  brief: string (what is this, why ship it)
  artifacts: Artifact[] (drafts, code, research docs)

  // Publishing
  publishTargets: ('linkedin' | 'blog' | 'render' | 'twitter' | 'email')[]
  publishedUrls: {target: string, url: string, publishedAt: timestamp}[]

  // Tracking
  lastActivity: timestamp
  completedAt?: timestamp
  archivedReason?: string
}

Artifact {
  id: string
  projectId: string
  type: 'draft' | 'code' | 'research_memo' | 'design'
  content: string
  generatedBy: 'human' | 'claude' | 'chatgpt' | 'perplexity' | string
  version: number
  status: 'draft' | 'ready' | 'published'
}
```

### 2. Capture Layer: Multiple Entry Points

**The goal: Zero friction to get ideas into the system.**

#### Capture Methods

1. **Chrome Extension / Bookmarklet**
   - Current tab URL + selected text + your note
   - One click ‚Üí idea captured
   - Extension can have quick classify: "Is this content/code/research?"

2. **Email-to-Inbox**
   - Forward emails to `ideas@ultrathink.yourdomain`
   - Subject becomes title
   - Body becomes context
   - Links auto-extracted

3. **CLI Tool**
   ```bash
   ultrathink capture "Idea about AI coordination"
   ultrathink capture --url="..." --note="..."
   ```

4. **API Endpoint**
   - POST /api/projects
   - Any tool can push ideas in
   - Your voice memo app, Slack, wherever

5. **Browser Tab Scraper** (power move)
   - CLI command: `ultrathink ingest-tabs`
   - Scrapes all open Chrome tabs
   - Creates project for each with auto-summary
   - Closes tabs (optional flag)
   - Forces you to process, not hoard

#### Auto-Processing on Capture

When an idea comes in, Ultrathink runs:

1. **LLM Classifier** (your choice: Claude API, local model, whatever)
   - Type: content/code/research
   - Tags: extract keywords
   - Impact score: 1-5 based on your past shipping patterns
   - Effort: hours/days/weeks

2. **Deduplication Check**
   - Embedding similarity to existing projects
   - Flags: "This looks similar to Project #47"
   - You decide: merge or keep separate

3. **Auto-Brief Generation** (optional)
   - "Here's a 3-sentence brief for this idea..."
   - You edit or approve

**Result:** Captured ideas arrive in your backlog **already triaged**.

### 3. Brain Layer: Constraints & Prioritization

This is the **forcing function**. Ultrathink enforces your rules.

#### Constraint Engine

Define rules once:

```yaml
constraints:
  max_active_projects: 3
  min_shipped_per_week: 1
  stale_after_days: 14

  auto_archive:
    - condition: "stale AND effort='weeks'"
      action: archive
      reason: "Too big, too stale. Break it down or kill it."

  weekly_review:
    - day: Sunday
    - time: "18:00"
    - template: "review_prompt.md"
```

#### Daily Focus Algorithm

Every morning (or on demand):

```
Ultrathink analyzes:
- Projects in ACTIVE state
- Projects in REVIEW state (you need to ship these)
- Constraint violations (3+ active? Nothing shipped this week?)

Outputs:
  "TODAY'S FOCUS"

  üî¥ MUST SHIP:
  - [Project #23: LinkedIn post on AI tools] ‚Üí Move to SHIPPED

  üü° MAKE PROGRESS:
  - [Project #15: Blog about orchestration] ‚Üí Add 500 words

  üü¢ IF TIME:
  - [Project #31: Research RAG architectures] ‚Üí Start brief

  ‚ö†Ô∏è  WARNINGS:
  - You have 4 active projects (max 3). Archive one or ship one.
  - No ships this week yet. Priority: ship something by EOD Friday.
```

You can override, but the system makes its expectations clear.

#### Priority Scoring

Auto-calculated:

```
priority = (impact_score * 10)
         + (days_since_capture * 0.5)
         - (effort_penalty)
         + (alignment_with_focus_areas)
```

You tweak the formula. System applies it.

### 4. Pipeline Layer: Pluggable Workflows

Instead of hardcoded "content agent ‚Üí research agent ‚Üí draft agent", think **workflows with your existing tools**.

#### Pipeline Architecture

```typescript
Pipeline {
  name: string
  trigger: 'manual' | 'auto_on_scope' | 'schedule'
  steps: Step[]
}

Step {
  name: string
  type: 'ai_prompt' | 'human_input' | 'api_call' | 'script'
  config: {
    // For AI prompts
    model?: 'claude' | 'gpt4' | 'perplexity'
    prompt?: string
    inputFrom?: 'project.brief' | 'artifact.content'
    outputTo?: 'create_artifact' | 'update_brief'

    // For scripts
    script?: string

    // For human input
    requestType?: 'approval' | 'input' | 'select'
  }
}
```

#### Example: Content Pipeline

```yaml
pipelines:
  - name: "LinkedIn Post"
    trigger: manual
    steps:
      - name: "Research"
        type: ai_prompt
        config:
          model: perplexity  # Uses Perplexity API for research
          prompt: "Research this topic: {{project.brief}}. Find 5 key insights and supporting data."
          outputTo: create_artifact
          artifactType: research_memo

      - name: "Generate Draft"
        type: ai_prompt
        config:
          model: claude
          prompt: |
            Using this research: {{artifact.research_memo.content}}
            Write a LinkedIn post about: {{project.title}}

            Voice: {{user.voice_guidelines}}
            Format: Hook ‚Üí Body ‚Üí CTA
            Length: 150-200 words
          outputTo: create_artifact
          artifactType: draft

      - name: "Human Review"
        type: human_input
        config:
          requestType: approval
          message: "Review draft and approve or provide feedback"

      - name: "Publish"
        type: api_call
        config:
          service: linkedin_publisher
          action: create_draft
```

#### Example: Code/Experiment Pipeline

```yaml
pipelines:
  - name: "Deploy to Render"
    trigger: manual
    steps:
      - name: "Generate Spec"
        type: ai_prompt
        config:
          model: claude
          prompt: "Turn this idea into a technical spec: {{project.brief}}"
          outputTo: create_artifact
          artifactType: spec

      - name: "Approve Spec"
        type: human_input
        config:
          requestType: approval

      - name: "Generate Code"
        type: ai_prompt
        config:
          model: claude
          prompt: "Implement this spec: {{artifact.spec.content}}. Generate a FastAPI app with a simple frontend."
          outputTo: create_artifact
          artifactType: code

      - name: "Create Repo"
        type: script
        config:
          script: "create_github_repo.sh"

      - name: "Deploy to Render"
        type: script
        config:
          script: "deploy_to_render.sh"

      - name: "Announce"
        type: ai_prompt
        config:
          model: claude
          prompt: "Write a LinkedIn post announcing this experiment: {{project.title}}. Include live URL: {{artifact.deployment.url}}"
          outputTo: create_artifact
          artifactType: announcement_draft
```

**Key Points:**
- Pipelines are YAML configs, not hardcoded
- You can use ANY AI tool (Claude, GPT-4, Perplexity, local models)
- You stay in the loop (human approval gates)
- Steps can be reordered, added, removed
- You can create new pipelines in 5 minutes

### 5. Publisher Layer: Multi-Destination Adapters

The "different publishing arms" you mentioned. Make this pluggable.

#### Publisher Interface

```typescript
interface Publisher {
  name: string
  publish(artifact: Artifact, config: PublishConfig): Promise<PublishResult>
}

PublishResult {
  success: boolean
  url?: string
  error?: string
}
```

#### Publisher Implementations

1. **LinkedIn Publisher**
   ```typescript
   // Option 1: Safe (via Buffer/Hypefury API)
   // Option 2: Direct (LinkedIn unofficial API - grey area)

   publish(artifact) {
     // Post content to LinkedIn
     // Return URL of published post
   }
   ```

2. **Blog Publisher**
   ```typescript

   // - Git-based markdown blog (commit + push)

     // Push to your blog
     // Trigger rebuild if static site
     // Return URL
   }
   ```

3. **Render Publisher**
   ```typescript
   publish(artifact) {
     // Create new Render service via API
     // Or deploy to existing service
     // Return live URL
   }
   ```

4. **Twitter Publisher**
   ```typescript
   publish(artifact) {
     // Thread support

   }
   ```

5. **Email/Newsletter Publisher**

## Key Architectural Decisions

### 1. Pluggable AI Backends
Don't lock yourself into one LLM. Use an adapter pattern:

```typescript
interface LLMProvider {
  complete(prompt: string, config: LLMConfig): Promise<string>
}

// Implementations: ClaudeProvider, GPTProvider, PerplexityProvider, OllamaProvider
```

You can switch providers per pipeline step.

### 2. Human-in-the-Loop by Default
Never let AI fully automate publishing. Always have approval gates. Trust is earned.

### 3. Data Portability
All project data in simple JSON/markdown. You can:
- Export everything
- Backup to Git
- Switch to a different system if needed

### 4. API-First Design
Build the API first. Frontend is just a client. Later you can:
- Add mobile app
- Add CLI
- Add Slack bot
- Let other tools integrate
